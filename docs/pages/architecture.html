What each layer owns

LayerOwnsMust not own
Pages / components UX, role-gated buttons, polling UX Service-role secrets, inventing authorization
Service adapter Mock vs Supabase routing, error fallback policy Business rules that belong in RPCs
RLS policies Row visibility (workspace membership) Complex multi-table state machines (use RPCs)
Guarded RPCs Writes that change status, enqueue jobs, create campaigns Long-running crawl execution
Workers Claim/lease jobs, fetch pages, DNS checks, publish results Customer auth decisions (trust the job row)

2. Generic request path (reads)

1
User opens a protected route

ProtectedRoute → useSeoAccess() resolves session, SEO module access, workspace, active website.

React Router /seo/*
↓
2
Page hook asks a service

e.g. auditService.fetchLatestRun(websiteId) via TanStack Query.

↓
3
runWithServiceAdapter()

If VITE_SEO_DATA_MODE=mock (default) → mock store. If supabase + configured → real client call.

↓
4
Supabase JS → PostgREST

select / rpc with the user’s JWT. Anon key alone is not enough — RLS uses auth.uid().

↓
5
Postgres returns only allowed rows

Workspace-member SELECT policies filter by membership. Non-members get empty / denied.

3. Trust boundaries

┌──────────────────────────────┐     JWT + anon key      ┌─────────────────────────────┐
│  Browser (untrusted)         │ ───────────────────────► │  Supabase / Postgres         │
│  React, mocks, local state   │                          │  RLS + SECURITY DEFINER RPCs │
└──────────────────────────────┘                          └──────────────┬──────────────┘
                                                                         │
                                         service_role (never in browser) │
                                                                         ▼
                                                          ┌─────────────────────────────┐
                                                          │  crawler-worker (trusted)    │
                                                          │  claim → crawl → publish     │
                                                          │  verify-once → DNS-TXT       │
                                                          └─────────────────────────────┘
The frontend may call customer RPCs (crawl request, approval transition, ownership initiate, …). It must never call service-role-only functions such as seo_crawl_claim_job or seo_ownership_verification_claim.

4. Tenancy model

auth.users ──1:1── profiles                    (Digibility Core, shared)
     │
     ├── user_module_access (module = 'seo' | 'visibility')
     ├── seo_subscriptions / seo_plan_limits / seo_usage_*   (schema; limits not fully enforced yet)
     │
     └── seo_workspace_members ──► seo_workspaces
                                      │
                                      └── seo_websites  ◄── EVERY SEO child record
                                              │
                                              ├── audits / recommendations / approvals
                                              ├── content studio
                                              ├── page inventory / performance / decline
                                              ├── authority + AI visibility
                                              ├── crawl_* + ownership_*

Invariant: operational SEO rows store both website_id (FK source of truth) and website_url (historical snapshot).
