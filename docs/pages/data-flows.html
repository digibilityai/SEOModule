Data-flow sequences

How data is created, transformed, and published across modules — from website onboarding through crawl publishing and Stage 6 workflows.

Flows
  1. Website as the data anchor
  2. Manual / seeded audit path
  3. Live crawl → audit / inventory publish
  4. Recommendations → approval
  5. Content Studio pipeline
  6. Page performance → decline diagnosis
  7. Off-page + AI visibility
  8. Ownership verification ledger

1. Website as the data anchor

seo_workspaces
   └── seo_websites  (website_url unique per workspace)
          ├── seo_business_onboarding     (1:1 context)
          ├── seo_connection_status      (integration placeholders)
          └── [all feature tables keyed by website_id + website_url snapshot]

Creating a website is the prerequisite for website-scoped routes (requireWebsite). Setup routes (/seo/websites and /seo/onboarding remain reachable without an active website.

2. Manual / seeded audit path (pre-crawler / demo)

seo_run_audit → seo_audit_runs (running) → seed / future worker fills seo_audit_issues → seo_recommendations

Historically seo_run_audit only created a running run; completion required a trusted backend. The crawler publishing path (below) is the real completion path for crawled sites.

3. Live crawl → audit / inventory publish

This is the most important end-to-end data sequence in the repo today.

1
Enqueue (customer)

seo_crawl_request_audit writes:

seo_audit_runs (running) + seo_crawl_jobs (queued) linked by audit_run_id.

↓
2
Claim + execute (worker)

Claim issues lease_token. Writes go to crawler-domain tables only at first:

seo_crawl_events seo_crawl_attempts seo_crawl_discovered_pages seo_crawl_sitemaps seo_crawl_page_snapshots seo_crawl_issues
↓
3
Publish (worker, transactional)

seo_crawl_worker_publish_results maps crawler findings into product tables:

seo_page_inventory seo_audit_issues seo_crawl_publications seo_audit_runs → completed

Uses deterministic 29-code map seo_crawl_issue_audit_map. Does not write recommendations or Page Performance snapshots.

↓
4
UI consumes existing services

Audit page + Page Performance page re-fetch inventory/issues; crawl panel shows terminal status + freshness.

Stale-job protection: an older crawl cannot overwrite newer inventory facts. Manual/user-owned inventory rows are preserved; missing pages are never auto-deleted as “removed”.

4. Recommendations → approval

seo_audit_issues
   └── seo_recommendations  (linked to audit_run / issue where relevant)
          └── seo_approval_items
                 ├── seo_approval_comments
                 └── seo_approval_activity   (append-only timeline)

Status changes for approval items flow only through seo_approval_transition.

5. Content Studio pipeline

opportunity → keyword plan → competitor summaries → wireframe → format inputs → draft + sections

Supporting tables: revisions, comments, activity, assets (metadata; bytes in private Storage bucket seo-content-assets). Transitions: seo_content_transition.

6. Page performance → decline diagnosis

seo_page_inventory
   ├── seo_page_keywords
   └── seo_page_performance_snapshots
          └── view: seo_page_performance_latest
                 └── seo_decline_diagnoses
                        ├── seo_decline_diagnosis_evidence
                        └── view: seo_decline_diagnoses_current
Locked module: Page Performance snapshot writes are not produced by the crawler. Snapshots are expected from GSC/GA4 later. Decline diagnosis can be created via seo_create_decline_diagnosis_from_snapshot (backend RPC).

7. Off-page + AI visibility

Authority

seo_authority_opportunities
        │  (status via seo_authority_opportunity_transition)
        ▼
seo_authority_campaigns  ←── seo_authority_campaign_create (atomic)
        ├── seo_authority_campaign_opportunities  (junction)
        ├── seo_authority_campaign_tasks
        └── seo_authority_activity                (append-only)

AI Visibility / GEO

seo_ai_prompt_tracking   (time-series visibility for prompts)
seo_ai_content_gaps
seo_ai_mentions          (brand / competitor mention rows)

Reads are wired; AI Visibility writes and real LLM ingestion remain deferred (preview / manual_seed data).

8. Ownership verification ledger

Customer path (no DNS yet):
  seo_ownership_verifications  ← initiate / recheck / reverify / revoke RPCs
  seo_ownership_verification_events  (append-only, customer-readable)

Worker path (DNS-TXT):
  seo_ownership_verification_claims  (internal lease ledger; not customer-readable)
    claim → resolve DNS → record_result → verification status verified|failed

Absence of a verification row = unverified. Challenge token is shown to the customer as the DNS TXT value; lease tokens never leave the worker/admin path.
