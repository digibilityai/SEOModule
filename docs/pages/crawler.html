Crawler pipeline

Hybrid control plane + dedicated Node worker. The browser never crawls; it only enqueues/cancels/observes. The worker owns fetch, discovery, extraction, issue detection, and publishing.

End-to-end sequence

1
Customer: Start crawl

crawlService → seo_crawl_request_audit → returns {auditRunId, crawlJobId}.

↓
2
Worker: claim

seo_crawl_claim_job with FOR UPDATE SKIP LOCKED; receives lease token; heartbeats while working.

↓
3
Discovery

Safe HTTP transport (DNS-rebinding checks), robots.txt, sitemap, same-origin <a> BFS under budgets → discovered pages + sitemaps tables.

↓
4
Extraction + issues

Facts from already-fetched HTML (title, headings, canonical, robots directives, hashes, …). Versioned rule registry → page + site duplicate issues.

↓
5
Publish

seo_crawl_worker_publish_results upserts inventory + audit issues, records publications, completes audit run. No recommendation writes.

↓
6
UI observes

Polling stops at terminal status; links to Audit results + Page Inventory. Cancel path finalizes linked running audits as failed when appropriate.

Status model (customer-visible)

Queued→ Preparing→ Crawling→ Completed / Partial / Failed / Cancelled

Also: Waiting to retry, Cancelling. Internal attempt diagnostics stay on seo_crawl_attempts (not customer UI).

Security guardrails

  • http/https only; block private/link-local/metadata ranges; re-validate after DNS and each redirect
  • Honor robots.txt (RFC 9309); XXE-safe sitemap parsing
  • Budgets: max pages/depth/timeout/response size; identifiable User-Agent
  • Lease-token ownership: stale workers cannot complete another’s job
  • Data minimization: no full HTML/scripts/JSON-LD stored in snapshots
